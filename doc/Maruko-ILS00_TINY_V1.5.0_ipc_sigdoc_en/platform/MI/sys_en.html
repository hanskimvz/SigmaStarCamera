    1. OVERVIEW¶

    1.1. Module Description¶

    MI_SYS is the foundation module for the entire MI system, and it provides the basis for the operation of other MI modules.

    Figure 1‑1 : MI_SYS System Framework

    As shown in Figure 1‑1, the main functions of the MI_SYS Overview are as follows:

    • Realize MI system initialization, MMA memory buffer pool management.

    • Provides the registration device nodes of each module, the general interface established by the proc system.

    • Provide s/he interface for each module to establish a binding relationship and manage the flow of data between the modules.

    • Provide s/he interface for each module to request MMA continuous physical memory, manage memory allocation, map virtual addresses, and reclaim memory.

    • Provides the interface of each module to establish the worker thread, manages the creation, operation, destruction of each module thread.

    The ko file corresponding to MI_SYS is mi_sys.ko, the library files are libmi_sys.so and libmi_sys.a, and the header files are mi_sys.h and mi_sys_datatype.h. Please refer to the API interface description.

    1.2. Document format constraints¶

    Body: for writing the body content of the document, where the writing of the snippet needs to be written in an equal-width font.

    Body and bold: For writing important content in the body of a document.

    Italics: for writing in the Tips section of the document.

    Italics and bold: For writing important content in the Tips section of the document.

    1.3. Keyword Description¶

    ID: Identity document, which means unique encoding.

    MI: SStar SDK Middle Interface, in this article, a similar structure of "MI_SYS" represents the MI SYS module, a simple "MI" refers to the entire SDK.

    Hex: Hexadecimal.

    Kernel Mode: refers to code that works in kernel environments and has control over the hardware that operates directly, such as functions and threads in ko.

    User Mode: refers to working in a User environment, such as customer applications, system calls, and so on.

    APP: Application, which refers primarily to an application that calls the MI API.

    API: Application Programming Interface.

    NVR: Network Video Recorder.

    DVR: Digital Video Recorder.

    XVR: A Video Recorder with any one or more functions combinations. Currently Sstar XVR consists of DVR and NVR.

    IPC: IP Camera.

    HW: Hardware.

    Dev: Device, represents for MI module device in this article, see section 1.4.1 for detail.

    Chn: Channel, represents a channel of MI module device, see section 1.4.1 for detail.

    Port: represents a port in the MI module device channel, see section 1.4.1 for detail.

    Soc: System on Chip.

    SocId: represents for specific Soc serial number, some platform use PCIE for multi-chip cascading, so SocId needs to be used to distinguish. Some interfaces support cross-Soc settings in MI_SYS, which have SocId parameters. As shown in Figure 1-3, bridging Soc1 and Soc2 by PCIE. Take Soc1 as Master to run Linux environment, MI SDK and customer service program. Take Soc as Slave to run Linux environment and MI SDK. You can choose to operate on Soc1 or Soc2 by calling the API carrying the SocId parameter.

    Figure 1‑2: PCIE cascaded Soc model

    Additional notes:

    • Without separate instructions, the MI_SYS in this article have the same meaning as SYS, MI_DISP, and DISP, and the remaining modules have similar names.

    • All module names that appear in this article can be found in MI_ModuleId_e.

    1.4. Flow frame¶

    1.4.1. Relationship between Dev/Chn/Port¶

    A typical MI module will have a Dev/Chn/Port three-stage structure as shown in the figure below.

    Dev

    An MI module will have one or more Devs, and in general different Devs indicate that the module device needs to invoke different HW resources or work in different working patterns. For example, VENC needs to invoke different HW resources when encoding H264/H265 and Jpeg, and Dev has to separate.

    Chn

    A Dev will have one or more Chns, which generally mean that the channel is different, and that the channel, although it shares HW resources with other Chns under Dev, is different from the data source or working pattern. For example, the source of the code flow is different, Chn is generally not the same.

    Port

    A Chn will have one or more ports, including Input Port (for data inflow) and Output Port (for data outflow). Different Ports indicate that the channel shares HW resources and data sources with other Ports under Dev and Chn, but the parameters that need to be set are different, such as different resolutions.

    In general, port is the smallest independent unit for the customer to operate the MI module because it identifies all the information: HW resources, data sources and parameter properties.

    Note: InputPort and OutputPort are not necessary in a chn, it depends on the behaviours of module.

    Figure 1‑3 : Three-level structure of the MI module

    Figure 1‑3 lists the different scenarios of Port arrangement under a single Chn in MI_SYS:

    1. InputPort only, which is required for data inflow, but no need for output module. Such as Disp, which display the results on the Panel directly.

    2. Output only, which doesnot need for data input, or input data without modules in MI_SYS. Such as VDEC, in which the data can be sent to Vdec RingPool directly by calling the VDEC interface.

    3. One InputPort and one OutputPort, which are used by modules for single data input and output. Such as LDC, data input first, and then hardware process to output result.

    4. One InputPort and more than one OutputPort, which are used for single data input and diffrent output process to output different data. Such as SCL, share a data source but output various results according diffrent OutputPort parameters.

    Tips:

    The boundaries between Chn and Port are not always clear, if there is a difference between the interpretation of a module's API documentation and the explanation above, call the module to its API documentation.

    1.4.2. A typical NVR data stream¶

    The following figure is a typical NVR data flow model. The flow process is as follows:

    1. Establishing a binding relationship with Vdec -Disp;

    2. User writes a stream to RingPool in Vdec;

    3. Vdec decoding, writing decoded data to Vdec OutputPort requested memory, sent to the next level;

    4. Disp receive the data and display it.

    Figure 1‑4: A typical NVR data stream model

    1.4.3. A typical IPC data stream¶

    The following figure is a typical IPC data flow model with the following flow procedures:

    1. Establishing a binding relationship with Vif-Isp-Scl-Venc;

    The common data stream scenarios are listed below to illustrate the working mechanism of the frame rate. The members in the figure are defined as follows:

    Tips:

    To ensure the ease of use of the interface, MI_SYS will automatically calibrate the frame rate of the previous output.

    If there is a difference between the u32SrcFrmrate set by user and the internal calculation value of MI_SYS, the calculated value will replace u32SrcFrmrate for frame rate control. But the calculation takes time, when the frame rate changes greatly, there will be 1-2 seconds of jitter.

    1.5.1. One-to-One Binding¶

    Figure 1‑8: Bind scene

    Corresponding to the output of the previous Chnport One-to-one binding of the next Chnport input scene, the frame rate control is determined by Bind Type.

    If it is in FRAME_BASE mode, the previous output frame rate control is effective, and the output frame rate is controlled to the target value, and the next input frame rate control is not effective. If it is REALTIME/HW_RING mode, the previous output frame rate control is not effective, then the next input frame rate is directly controlled to the target value.

    Take the figure above as an example:

    Case1: set Bind FRC: (30, 5), Bind Type:FRAME_BASE, the output frame rate of the previous ISP is controlled to 5FPS.

    Case2: set Bind FRC: (30,5) Bind Type:REALTIME, the output frame rate of the previous ISP is maintained at 30FPS, and the input frame rate of the back-end SCL is controlled to 5FPS.

    Tips:

    Regardless of the scene in REALTIME/HW_RING mode, the previous output frame rate control is not effective, and only the next input frame rate can be controlled, which is determined by the hardware.

    1.5.2. One-to-Multiple Binding¶

    Figure 1‑9: Bind scene

    Corresponding to the output of the previous Chnport One-to-Multiple binding of the next Chnport input scene, the frame rate control is determined by Bind Type and Bind FRC.

    If it is in FRAME_BASE mode, the front-level ISP binds multiple back-levels at the same time, first control the frame rate to meet the frame rate requirements of all the back-level bind modules, and then perform frame rate control separately at the back-level output.

    If it is in REALTIME/HW_RING mode, the frame rate control of the previous output will not take effect, and the input of the subsequent stage will control the frame rate separately.

    Take the figure above as an example:

    In this scenario, the two bind frame rates for ISP ChnPort0 are (30, 5) (30, 10), so the Output frame rate of ISP ChnPort0 is 10FPS.

    The input of SCL ChnPort0 needs to control the frame rate from 10FPS to 5FPS. SCL ChnPort1 does not require frame rate control.

    1.5.3. Set UserDepth While Binding¶

    Figure 1‑10: Set UserDepth While Binding

    The scene of UserDepth is also set in the front output when Bind the rear stage, and the frame rate control is fixed and unique.

    When user needs to use the output data, UserDepth set by MI interface will be called, and the previous Output will give user the latest data, so the previous output does not control the frame rate.

    If there is a binding at the rear stage, do frame rate control at the input port. If not, the frame rate control will not take effect. At this time, the data output is determined by the speed of data generation and consumption. Explain in detail when explaining the concept of Depth.

    Take the figure above as an example:

    In this scenario, the two bind frame rates for ISP ChnPort0 are (30, 5), (30, 10), but the Output Depth with ISP ChnPort0 set is (1, 4), and the Output frame rate of ISP ChnPort0 is 30FPS. The input port of SCL ChnPort0 controls the frame rate from 30FPS to 5FPS. SCL ChnPort1 controls the frame rate from 30FPS to 10FPS.

    1.5.4. Supplement to VDEC Frame Rate Control in NVR Scenarios¶

    Figure 1‑11: Supplement to VDEC Frame Rate Control in NVR Scenarios

    Corresponding to Vdec Chnport's Output one-to-one bind post-stage Chnport Input scene, the frame rate control is determined by BindFRC.

    NVR requires user to input the code stream to Vdec, and it is difficult to ensure a balanced transmission speed. This is different from IPC, whose data source is strictly controlled by interrupts. If the stability of the incoming frame rate cannot be ensured, you can use the function of automatically calculating the sending frame rate provided by MI, which is automatically enabled when u32SrcFrmrate is set to -1, and MI will decide whether to allocate Buffer to Vdec according to u32DstFrmrate.

    NVR provides a double-speed playback function. When the Timing of the subsequent DISP cannot keep up, it needs to actively drop frames before inputting to the DISP. If VDEC decoding speed is faster than DISP display Timing, you can use the proportional drop frame function provided by MI to set the ratio of u32SrcFrmrate and u32DstFrmrate.

    Take the figure above as an example:

    Case1: set Bind FRC: (-1, 30) Bind Type: FRAME_BASE. Suppose the output FrameRate of the previous Vdec is 30, it takes 1000ms/30 = 33.3ms for the Vdec Output to get the buffer allocated by MI.

    Case2: set Bind FRC: (60, 30) Bind Type:FRAME_BASE , the ratio is 60/30 = 2/1. Suppose the output FrameRate of the previous Vdec is 60, only one of the two images decoded by the Vdec Output will be sent to Disp Input.

    Tips:

    NVR products can query the current ChnPort legacy frames through the API interface of MI_VDEC to dynamically select two strategies to achieve better results

    The function of automatically calibrating the real frame rate of previous Output when Vdec Bind is in the next stage is no longer effective.

      #define ST_DEFAULT_SOC_ID    0

    return MI_SUCCESS;

        }
    }
}

    close(fd);

}

    4. ERROR CODE¶

    4.1. The composition of the error code¶

    Mi returns the error code for 4 bytes, a total of 32bits, consisting of five parts as shown in Table 4-1:

    Table 4‑1: The composition of the error code

    Position Bits Description
    BIT [0-11] 12bits The error type that indicates the specific error meaning of the error code.
    BIT [12-15] 4bits Error level, fixed return 2.
    BIT [16-23] 8bits Module ID, which module the error code belongs to.
    BIT [24-31] 8bits Fixed to 0xA0.

    Tips:

    We can simply think of the error code as two double bytes (16bits) for quick interpretation, in the case of the error code 0xA0092001:

    "A009": Indicates that the error occurred in a module with module ID 9. By looking at the definition of "MI_ModuleId_e", it can be known that it is the MI_SYS module.

    "2001": The error type that indicates the error is 1, which means "Device ID is out of legal range".
